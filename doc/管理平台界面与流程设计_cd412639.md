# AutoMarketing 管理平台界面设计与操作流程规划

## 一、整体导航结构

管理平台采用左侧菜单导航 + 顶部面包屑的布局，导航层级如下：

```
AutoMarketing 管理后台
├── 仪表盘（Dashboard）
├── 任务中心（Task）
│   ├── 任务列表
│   ├── 创建任务
│   └── 任务详情（隐藏菜单，从列表进入）
├── 数据中心（Data）
│   ├── 触达用户
│   └── 内容数据
├── 配置中心（Config）
│   ├── 平台管理
│   ├── 业务线管理
│   ├── 关键词管理
│   ├── 提示词管理
│   └── AI模型配置
├── 统计分析（Stats）
│   ├── 数据概览
│   └── 趋势分析
├── 系统管理（System）
│   ├── 账号配置
│   └── 操作日志
```

**操作流程：**
1. 进入任务列表页，默认展示最近7天的任务，按创建时间倒序
2. 通过顶部筛选栏可按任务类型（爬虫/私信/评论回复）、业务线、状态（pending/queued/running/success/failed/cancelled）筛选
3. 支持日期范围筛选
4. 每行操作列提供：查看详情、启动（仅 pending 状态）、停止（queued 或 running 状态）、重试（failed 或 cancelled 状态）、删除（仅非 running 状态）
5. 点击右上角「创建任务」按钮跳转到创建任务页

### 2.3 任务中心 - 创建任务

**界面布局：**

```
+------------------------------------------------------------------+
| 创建任务                                                            |
+------------------------------------------------------------------+
| 步骤条:  ① 选择类型  -->  ② 配置参数  -->  ③ 确认启动              |
+------------------------------------------------------------------+
|                                                                     |
|  步骤一：选择任务类型                                                 |
|  +------------+  +------------+  +--------------+                  |
|  |  关键词爬虫  |  |  私信触达   |  |  评论自动回复  |                |
|  |  搜索关键词  |  |  向筛选出   |  |  自动回复目标  |                |
|  |  爬取评论   |  |  的用户发送  |  |  用户的评论   |                |
|  |            |  |  个性化私信  |  |              |                |
|  +------------+  +------------+  +--------------+                  |
|                                                                     |
|  步骤二：配置参数（以爬虫任务为例）                                      |
|  +--------------------------------------------------------------+ |
|  |  业务线:    [TikTok / golf           v]                        | |
|  |  关键词:    [golf simulator] [indoor golf] [+ 添加]            | |
|  |            (从关键词配置中勾选 或手动输入)                         | |
|  |  内容类型:  [x] 视频  [x] 评论  [ ] 帖子                       | |
|  |  最大采集:  [50] 条/关键词                                      | |
|  |  AI筛选:    [x] 启用AI意图筛选                                  | |
|  |  筛选提示词: [客户筛选-高尔夫 v] (自动关联业务线激活的提示词)        | |
|  |  排除作者:  [x] 排除视频作者                                     | |
|  +--------------------------------------------------------------+ |
|                                                                     |
|  步骤二（私信任务）：                                                 |
|  +--------------------------------------------------------------+ |
|  |  业务线:      [TikTok / stock          v]                      | |
|  |  目标用户:    [从筛选结果中选择]  已选: 45人                      | |
|  |  消息模式:    (*) 个性化生成  ( ) 固定话术                       | |
|  |  生成提示词:  [私信生成-金融投资 v]                               | |
|  |  发送上限:    [50] 条                                            | |
|  |  发送间隔:    [8-20] 分钟（防封号）                               | |
|  |  账号选择:    [NEAGLE_GOLF v] (指纹浏览器账号)                    | |
|  +--------------------------------------------------------------+ |
|                                                                     |
|  步骤三：确认                                                        |
|  +--------------------------------------------------------------+ |
|  |  任务摘要:                                                      | |
|  |  - 类型: 关键词爬虫                                              | |
|  |  - 业务线: TikTok > golf                                        | |
|  |  - 关键词: golf simulator, indoor golf (2个)                    | |
|  |  - 预计采集: 100条视频                                           | |
|  |  - AI筛选: 启用                                                 | |
|  |                                                                | |
|  |                     [取消]  [保存为模板]  [立即启动]               | |
|  +--------------------------------------------------------------+ |
+------------------------------------------------------------------+
```

**操作流程：**
1. 选择任务类型：关键词爬虫 / 私信触达 / 评论自动回复
2. 配置参数：
   - **爬虫任务**：选择业务线 -> 勾选或输入关键词 -> 选择内容类型 -> 设置采集数量 -> 选择是否启用AI筛选
   - **私信任务**：选择业务线 -> 从AI筛选后的潜在客户列表中选择目标用户 -> 选择消息模式（个性化生成/固定话术） -> 设置发送上限和间隔
   - **评论回复任务**：选择业务线 -> 选择目标视频和评论 -> 配置回复提示词
3. 确认信息无误后点击「立即启动」，任务状态置为 pending 并随即置为 queued（加入执行队列），由独立 Worker 进程认领后自动执行（queued → running → success/failed）；也可先保存为 pending，稍后在任务列表点「启动」再入队。详见「六、任务执行引擎（Worker）设计」。

## 六、任务执行引擎（Worker）设计

### 6.1 总体架构

采用**独立 Worker 进程 + SQLite 作为任务队列**的方案，不引入 Redis/Celery，降低部署复杂度。

```
+-----------------+        写 status='queued'         +------------------+
|  管理平台前端     |  --点击「启动」--> POST /start -->  |   FastAPI (API)   |
|  (naive-ui)     |                                    |  run.py           |
+-----------------+                                    +---------+--------+
        ^                                                        |
        | 每 5s 轮询任务详情/日志                                    | UPDATE task_executions
        |                                                        v
+-------+---------+        轮询 status='queued'        +------------------+
|  任务详情页       |  <--读 progress/task_logs--        |   SQLite DB       |
|  (进度条/日志)    |                                    |  automarketing.db |
+-----------------+                                    +---------+--------+
                                                                 ^
                                        认领 queued->running       | 写 contents/contacts
                                        写 progress/task_logs      | 更新 progress/日志
                                                                 |
                                                       +---------+--------+
                                                       |  独立 Worker 进程  |
                                                       |  worker.py        |
                                                       |  src/worker/*     |
                                                       |  (Playwright+AI)  |
                                                       +------------------+
```

- **API 与 Worker 完全解耦**：API 只负责改状态（入队），Worker 只负责捞取执行；两者共用同一 SQLite 与 `src/core/database.py`。
- **前端零改造实时性**：Worker 每步写 `task_executions.progress` 与 `task_logs`，前端详情页在 `queued/running` 期间每 5s 轮询自动刷新。

### 6.2 任务状态机

```
pending ──(点击启动 / POST /start)──> queued ──(Worker 认领)──> running ──┬──> success
   │                                    │                        │       ├──> failed（异常）
   └─────────────(可直接删除)────────────┘                        │       └──> cancelled（手动停止）
                                     (可停止)                  (可停止)
```

| 状态 | 含义 | 触发 |
|------|------|------|
| pending | 已创建，未入队 | 创建任务 |
| queued | 已加入执行队列，等待 Worker 认领 | `POST /tasks/{id}/start` |
| running | Worker 已认领，执行中（写 start_time） | Worker 认领 |
| success | 执行成功（progress=100，写 end_time） | Worker 正常结束 |
| failed | 执行异常（写 error_message、end_time） | Worker 捕获异常 |
| cancelled | 手动停止（写 end_time） | `POST /tasks/{id}/stop` |

- 停止：允许对 `queued` 或 `running` 的任务置 `cancelled`；Worker 在**步骤之间**调用 `is_cancelled()` 检测到 `cancelled` 后中断并清理（关闭浏览器）。

### 6.3 SQLite 作为队列的机制

- **入队**：`POST /tasks/{id}/start` 将 `pending` 任务置 `queued`。
- **认领**：Worker 轮询 `SELECT * FROM task_executions WHERE status='queued' ORDER BY created_at ASC LIMIT 1`，随后用乐观锁 `UPDATE ... SET status='running' WHERE id=:id AND status='queued'` 认领；`rowcount=0` 表示被抢占则跳过。
- **串行执行**：并发度=1，同一时刻仅一个任务运行，保护浏览器账号不被风控。
- **实时进度**：每次更新使用独立 `SessionLocal` 会话并立即 `commit`，避免长事务占用连接、保证前端可见。

### 6.4 模块清单

| 文件 | 职责 |
|------|------|
| `worker.py` | 项目根入口，加载 .env 后调用 `src.worker.runner.main()`；启动方式 `python worker.py` |
| `src/worker/runner.py` | 主轮询循环、状态机、任务分发、DB 工具方法（`TaskContext.log/update_progress/is_cancelled`）、优雅退出（SIGINT/SIGTERM） |
| `src/worker/config.py` | 解析 Chrome profile 目录、`headless`（env `WORKER_HEADLESS`）、DeepSeek Key（`conf/api_key.json`）、轮询间隔（env `WORKER_POLL_INTERVAL`） |
| `src/worker/pipelines/tiktok_scrape.py` | tiktok + scrape 的写库版执行流程 |

**Chrome profile 解析规则**（`resolve_chrome_profile`）：
1. 优先取 `business_lines.config.chrome_user_data_dir`；
2. 否则取平台默认常量（tiktok → `chrome_data/Chrome_Bot_Data_TK`）；
3. 相对路径统一相对项目根解析为绝对路径。

### 6.5 爬虫任务（scrape）完整执行流程

数据来源：`task_executions.task_config`（keywords / max_items_per_keyword / ai_filter_enabled / ai_prompt_template_id / exclude_author）、`prompt_templates.template_content`。

```
步骤1: Worker 认领任务，queued -> running，写 start_time + 日志「Worker 已认领任务」
步骤2: 解析配置与 Chrome profile，launch_persistent_context 打开浏览器，日志「任务启动，打开浏览器」
步骤3: 逐个关键词搜索视频（search_keywords），按 max_items_per_keyword 截断、去重
        > 视频写入 contents（INSERT OR IGNORE，content_type='video'，记录 title/author/source_keyword/url）
        > 日志「关键词 X 搜索完成，获取 N 个视频」
步骤4: batch_scrape_comments 批量爬评论；汇总待分析评论（去重 + 按 exclude_author 过滤作者）
        > 回填 update_progress(total=待分析评论数)，日志「评论爬取完成，共 N 条待分析评论」
步骤5: 逐条评论 AI 筛选（若 ai_filter_enabled）：
        > 取 prompt_templates.template_content 组装 prompt（追加标题+评论）
        > 调用 get_text_response_ds（asyncio.to_thread 避免阻塞事件循环）判定 yes/no
        > 命中者写入 contacts（INSERT OR IGNORE，contact_status='pending'，metadata 存评论/来源视频/命中关键词）
          并写 contact_interactions（interaction_type='scraped'，关联 task_execution_id）
        > 每条更新 success/failed/pending/progress；每 10 条写一次阶段日志
步骤6: 结束，日志「分析完成：已分析 X 条，异常 Y 条，命中潜在客户 Z 个」，progress=100，running -> success
```

**进度语义**：`total_items`=实际待分析评论数（爬完评论后回填）、`success_items`=已分析评论数、`failed_items`=分析异常数、`pending_items`=剩余；命中潜在客户数记录在日志与 `contacts` 表。

**停止与失败处理**：步骤间 `is_cancelled()` 命中则关闭浏览器并 return（状态保持 cancelled）；顶层 try/except 捕获异常置 `failed` + `error_message`。

### 6.6 私信任务（message）完整实现流程规范（后续开发依据）
